## 面试问题整理

### 基础与架构

- **数据结构**：谈谈你掌握的数据结构有哪些？
<details>
<summary>查看答案</summary>

常见数据结构包括：
1. **线性结构**：数组（Array）、链表（LinkedList）、栈（Stack）、队列（Queue）。
2. **非线性结构**：哈希表（HashMap）、树（二叉树、红黑树、B+树）、图（Graph）。
3. **高级结构**：跳表（SkipList）、位图（BitMap）、布隆过滤器（Bloom Filter）。
</details>

- **一致性哈希**：请讲一下一致性哈希的原理和解决的问题。
<details>
<summary>查看答案</summary>

**原理**：将整个哈希值空间组织成一个虚拟的圆环（0 - 2^32-1），节点和数据都映射到环上。数据按顺时针方向寻找最近的节点进行存储。
**解决的问题**：解决传统哈希（hash(key) % N）在增加或删除节点时导致的大规模数据迁移问题（雪崩效应）。一致性哈希只需迁移邻近节点的数据。
**优化**：引入**虚拟节点**解决负载不均衡（数据倾斜）问题。
</details>

- **Ring Buffer**：请讲一下 Ring buffer（环形缓冲区）的相关内容。
<details>
<summary>查看答案</summary>

**概念**：一种首尾相连的固定大小缓冲区，使用两个指针（Head/Tail）标记读写位置。
**优点**：
1. **无需动态分配内存**：避免频繁的 GC 或内存碎片。
2. **锁无关（Lock-free）**：在单生产者单消费者模式下不需要加锁。
3. **高性能**：适用于高频数据交换，如 LMAX Disruptor 框架。
</details>

- **代理技术**：请讲一下反向代理的原理和常用工具，以及正向代理的相关内容。
<details>
<summary>查看答案</summary>

**正向代理**：代理**客户端**。客户端通过代理服务器访问目标服务器（如 VPN、梯子）。目标服务器不知道真实客户端 IP。
**反向代理**：代理**服务端**。客户端访问代理服务器，由代理服务器转发给后端节点（如 Nginx、HAProxy）。客户端不知道后端真实 IP。
**作用**：负载均衡、安全防护（隐藏真实服务器）、缓存静态资源。
</details>

### 网络与负载均衡

- **HTTP 状态码**：请讲解一下 HTTP 状态码的相关分类和含义（如 2xx、3xx、4xx、5xx）。
<details>
<summary>查看答案</summary>

- **1xx**：信息性状态码，表示请求已接收，正在处理。
- **2xx**：成功。200 (OK)、204 (No Content)、206 (Partial Content)。
- **3xx**：重定向。301 (永久)、302 (临时)、304 (命中缓存)。
- **4xx**：客户端错误。400 (语法错误)、401 (未授权)、403 (拒绝访问)、404 (找不到资源)。
- **5xx**：服务端错误。500 (内部错误)、502 (网关错误)、503 (服务过载)、504 (超时)。
</details>

- **负载均衡**：请讲解一下四层网络负载均衡和七层网络负载均衡的区别，以及各自的应用场景。
<details>
<summary>查看答案</summary>

- **四层负载均衡 (L4)**：基于 **传输层**（IP + Port）。通过修改数据包的目标 IP 地址转发。性能极高，但不理解应用层协议（如 HTTP URL）。场景：LVS、F5。
- **七层负载均衡 (L7)**：基于 **应用层**（HTTP、URL、Cookie、Header）。可以根据 URL 路径转发到不同服务器。功能丰富（动静分离），但性能略低于 L4。场景：Nginx、HAProxy。
</details>

- **连接机制**：长连接的适用场景是什么？哪些场景不适合使用长连接，原因是什么？
<details>
<summary>查看答案</summary>

**适用场景**：实时性要求高、频繁收发小数据的场景（如即时通讯、在线游戏、推送系统、数据库连接）。
**不适用场景**：
1. **用户量极大但访问稀疏**：如普通网页浏览。长连接会长时间占用服务器内存和端口资源，导致连接耗尽。
2. **原因**：维护 TCP 连接需要内存状态和心跳检测，资源开销大。
</details>

### 进程与中间件

- **IPC**：进程间通信的方式有哪些？哪种方式效率更高，为什么？
<details>
<summary>查看答案</summary>

**方式**：管道（Pipe）、消息队列（Message Queue）、信号量（Semaphore）、信号（Signal）、共享内存（Shared Memory）、套接字（Socket）。
**最高效**：**共享内存**。
**原因**：它不需要在内核和用户空间之间多次拷贝数据，多个进程直接映射同一物理内存块进行读写。
</details>

- **消息队列**：你了解的消息中间件有哪些？RabbitMQ、RocketMQ、Kafka 这三种消息中间件的区别是什么？
<details>
<summary>查看答案</summary>

- **RabbitMQ**：Erlang 开发，延迟极低（微秒级），功能丰富，但吞吐量相对较低。
- **RocketMQ**：阿里开源，Java 开发，高可靠、高吞吐，支持事务消息、定时消息，适合业务系统。
- **Kafka**：Scala/Java 开发，超高吞吐量（百万级），基于磁盘顺序读写和零拷贝，适合日志采集、大数据流处理。
</details>

- **序列化**：你在项目中使用过的序列化协议是什么，为什么选择该协议？
<details>
<summary>查看答案</summary>

**常用协议**：JSON, Protobuf, Hessian, Thrift。
**推荐选择**：**Protobuf** (Protocol Buffers)。
**原因**：1. **效率极高**：二进制编码，体积小、解析快。2. **跨语言支持**。3. **强类型定义**：利于维护。
</details>

- **服务发现**：你在项目中使用过的服务发现组件是什么，它的实现原理是什么？
<details>
<summary>查看答案</summary>

**常用组件**：Nacos, Consul, ETCD, Eureka。
**以 ETCD 为例**：
1. **原理**：基于 **Raft 一致性算法** 保证强一致性存储。
2. **流程**：服务启动时向注册中心注册租约并定期续约（心跳）；客户端监听（Watch）特定前缀的变更，实时获取最新的服务节点列表。
</details>

### 数据库与缓存

- **MySQL 主从**：请讲一下 MySQL 主从复制的实现原理（基于 binlog、redolog 相关）。
<details>
<summary>查看答案</summary>

1. **主库**：将变更操作写入 **binlog** (二进制日志)。
2. **从库 I/O 线程**：连接主库，读取 binlog 内容并写入本地的 **relay log** (中继日志)。
3. **从库 SQL 线程**：读取 relay log 并重放 SQL 语句，实现数据同步。
*注：redolog 是物理日志，用于主库自身的崩溃恢复（WAL 机制），不直接参与主从同步。*
</details>

- **数据一致性**：多个从节点之间出现数据不一致的问题该如何解决？
<details>
<summary>查看答案</summary>

1. **半同步复制 (Semi-Sync)**：至少一个从库接收到 binlog 后才返回写入成功。
2. **GTID (全局事务标识符)**：确保事务在集群中唯一且有序，简化故障恢复。
3. **读写分离中间件强制读主**：对于实时性要求极高的读取，直接强制路由到主库。
4. **数据校验工具**：定期使用 `pt-table-checksum` 进行一致性扫描。
</details>

- **Redis 基础**：Redis 中最常用的数据结构有哪些？
<details>
<summary>查看答案</summary>

**五大基础类型**：String, List, Hash, Set, ZSet (Sorted Set)。
**高级类型**：HyperLogLog (基数统计), Geo (地理位置), Bitmap (位图), Stream (消息流)。
</details>

- **ZSet 底层**：请讲一下 Redis 中 ZSet (sorted set) 的底层实现和优化策略。
<details>
<summary>查看答案</summary>

**实现**：**压缩列表 (ziplist/listpack)** 或 **跳表 (skiplist) + 字典 (dict)**。
- 当元素数量少且长度小时，用压缩列表节省空间。
- 否则升级为跳表。跳表负责范围查询，字典负责 $O(1)$ 时间查找元素分数。
</details>

- **哈希性能**：什么是小哈希和大哈希，二者在查找、插入性能上有什么区别？
<details>
<summary>查看答案</summary>

- **小哈希**：底层使用 **ziplist/listpack**。它是内存连续的存储，虽然查找是 $O(N)$，但因为数据量极小且 CPU 缓存友好，在小范围内比哈希表快且省内存。
- **大哈希**：底层使用 **hashtable**。标准 $O(1)$ 查找，但由于指针多、非连续存储，会有更多的内存碎片和元数据开销。
</details>

### 分布式与设计

- **分布式事务**：请讲一下 TCC 分布式事务算法的相关内容，以及它和 2PC、3PC 的区别。
<details>
<summary>查看答案</summary>

**TCC (Try-Confirm-Cancel)**：
1. **Try**：检查及资源预留。
2. **Confirm**：确认执行（幂等操作）。
3. **Cancel**：取消并释放资源（回滚）。
**区别**：
- **2PC/3PC**：属于**资源层**的强一致性方案，依赖数据库 XA 事务，性能差、容易阻塞。
- **TCC**：属于**应用层**的最终一致性方案，性能高，但业务侵入性强（每个接口都要写三套逻辑）。
</details>
