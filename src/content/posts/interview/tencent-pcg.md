## 面试问题整理

### 开源深挖
- **项目架构与 LLM 抽象**：介绍 AI 项目的整体架构。为什么对 LLM 调用做一层抽象？
  <details>
  <summary>查看答案</summary>

  > **架构**：通常分为接入层（API/网关）、业务逻辑层（Agent/Workflow编排）、基础设施层（大模型API、向量库、传统DB）。
  > **LLM抽象原因**：1. **解耦**：防止与单一厂商（如 OpenAI）强绑定。2. **统一接口**：兼容多模型（GPT, Claude, DeepSeek）的不同参数格式。3. **统一监控与计费**：便于拦截、重试、限流和 Token 统计。

  </details>

- **向量数据库**：项目里为什么需要向量数据库？主要用来做什么？检索一般是怎么做的？为什么选择这个向量数据库？
  <details>
  <summary>查看答案</summary>

  > **需求与用途**：大模型存在上下文长度限制和知识截止问题。向量库用于存储和检索外部私有知识库（RAG 架构），解决大模型幻觉。
  > **检索做法**：将 Query 经过 Embedding 模型转化为高维向量，在向量库中通过 KNN 或 ANN（如 HNSW、IVF）算法计算余弦相似度或内积，召回 Top-K 的文档块。
  > **选型依据**：根据数据量、检索延迟、社区活跃度、是否需要混合检索（Dense+Sparse）来选择，如 Milvus、Qdrant、Faiss 或 Elasticsearch（带向量支持）。

  </details>

### 实习项目
- **系统介绍**：介绍实习做的系统，主要解决什么业务问题？业务流程是怎样的？
  <details>
  <summary>查看答案</summary>

  > (因人而异，建议采用 STAR 法则：情境、任务、行动、结果来描述，突出系统的高并发、解耦或提效价值)

  </details>
- **架构解耦**：为什么要做模块解耦？原系统是怎样耦合的？为什么不用消息队列？
  <details>
  <summary>查看答案</summary>

  > **解耦原因**：提高系统的可维护性、可扩展性，隔离故障，便于独立部署和缩扩容。
  > **耦合举例**：比如下单后直接同步调用积分、物流、短信等服务，一个服务卡顿导致整个下单链路超时。
  > **为什么不用 MQ**：可能因为当时业务规模较小，引入 MQ 会增加系统复杂度（需处理消息丢失、重复消费等问题），所以选择了基于事件发布订阅（如 Spring Event）的本地解耦方案。

  </details>
- **异步任务**：异步任务是怎么处理的？失败了怎么办？有没有做重试机制？
  <details>
  <summary>查看答案</summary>

  > **处理方式**：通过线程池（\@Async）或定时任务框架（如 XXL-Job）处理。
  > **失败与重试**：如果任务失败，首先记录异常日志，并保存失败状态。可引入 **重试机制**：本地可用 \`@Retryable\`，或通过延迟队列、定时任务扫描失败记录进行指数退避重试。重试达到最大次数后转入死信队列或人工告警介入。

  </details>

### 缓存与高并发
- **缓存三剑客**：Redis 常见缓存问题有哪些？如何解决穿透和击穿？
  <details>
  <summary>查看答案</summary>

  > - **雪崩**：大量 Key 同时过期或 Redis 宕机。**解决**：过期时间加随机值、高可用集群（哨兵/Cluster）。
  > - **穿透**：查不存在的数据。**解决**：布隆过滤器、缓存空值。
  > - **击穿**：热点 Key 突然过期。**解决**：互斥锁（SETNX，只放一个请求去 DB）、逻辑过期（后台线程异步刷新）。

  </details>
- **防超卖与 Lua**：如何防止库存超卖？为什么使用 Lua 脚本？它解决了什么问题？
  <details>
  <summary>查看答案</summary>

  > **防超卖**：可以使用数据库乐观锁（\`WHERE stock > 0\`）或 Redis 分布式锁/Lua 脚本。
  > **Lua 脚本的作用**：将查询库存、判断库存是否充足、扣减库存这几步操作打包，Redis 会把整个 Lua 脚本作为一个整体执行，保证了多条命令的**原子性**，避免了并发下的数据不一致问题。
  > **注意**：Redis 的 Lua 脚本不具备关系型数据库那种回滚（Rollback）能力，如果脚本中途出错，已经执行的命令不会撤销。

  </details>
- **Redis 结构**：Redis 用过哪些数据结构？谈谈底层实现。
  <details>
  <summary>查看答案</summary>

  > String (SDS简单动态字符串), Hash (ziplist/hashtable), List (quicklist), Set (intset/hashtable), ZSet (ziplist/skiplist)。
  > （注：Redis 7.0 之后 ziplist 逐步被 listpack 替代）。

  </details>

### AI 功能相关
- **Function Calling**：大模型如何决定调用哪个函数？
  <details>
  <summary>查看答案</summary>

  > 我们在发起请求时，将系统可用的所有函数签名（名称、描述、参数结构）以 JSON Schema 格式和 prompt 一起发给 LLM。
  > 大模型内部经过训练，能够根据用户的 query 理解意图。如果它认为需要调用某个函数，就不会直接返回文本，而是返回一段包含函数名和入参 JSON 的特殊结构指令，由客户端执行后，再将结果喂给大模型生成最终回复。

  </details>
- **智能客服重构**：如果重构智能客服，会考虑怎么做？
  <details>
  <summary>查看答案</summary>

  > 1. **意图路由**：先用小模型或规则判断用户的意图（是闲聊、查订单还是技术支持）。
  > 2. **RAG 增强**：如果涉及业务知识，利用向量数据库和 BM25 混合检索企业知识库，将相关片段作为上下文注入。
  > 3. **Function Calling/Agent**：赋予客服调用内部 API 的能力，比如查询订单状态、发起退款等，实现真正的任务闭环。

  </details>
- **意图识别**：如何做用户意图识别？
  <details>
  <summary>查看答案</summary>

  > 1. **Prompt Engineering**：给出少样本（Few-shot）示例，让 LLM 做多分类输出。
  > 2. **微调 (Fine-tuning)**：使用业务数据对开源小模型（如 Llama3-8B）进行指令微调，成本低且响应快。
  > 3. **传统 NLP 结合**：使用 BERT 类模型进行文本分类，或者使用正则表达式处理极高频的固定问法。

  </details>

### Redis 进阶
- **持久化机制**：Redis 有哪些持久化方式？RDB 和 AOF 有什么区别？AOF 文件过大怎么办？重启数据是怎么恢复的？
  <details>
  <summary>查看答案</summary>

  > **方式与区别**：
  > 1. **RDB (Redis Database)**：定时生成内存快照（二进制文件）。恢复快，文件小，但容易丢失最后一次快照后的数据。
  > 2. **AOF (Append Only File)**：记录每一次写操作命令。数据安全性高（可配 everysec），但文件大，恢复慢。
  > **AOF Rewrite**：当 AOF 文件过大时，Redis 会在后台 fork 子进程，读取当前内存状态，生成新的、精简的写命令替换旧文件。
  > **恢复过程**：重启时，如果开启了 AOF，优先使用 AOF 恢复（因为数据更完整）；否则使用 RDB 恢复。Redis 4.0 后支持混合持久化（AOF 前半段写 RDB，后半段写增量 AOF），兼顾速度与安全。

  </details>
